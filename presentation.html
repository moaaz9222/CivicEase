CivicEase AI04 — Agent Deep Dive
Safety & Responsible AI

Four Layers of Programmatic Guardrails

L1
Input Schema Validation — Pydantic v2
Every LLM output is parsed against a strict IntakeProfile schema before any downstream agent sees it. Invalid or hallucinated fields trigger a safe fallback — never a downstream failure.
L2
Output Schema Enforcement — Agent 2 & 3
Policy assessments and action plans are validated against PolicyEvaluationResponse and ActionPlanOutput schemas. Wrong types, missing keys, or hallucinated fields return pre-defined safe fallbacks.
L3
URL Sanitization — Python + JavaScript
All LLM-generated URLs are verified by urllib.parse on the server and new URL() on the client. Only valid http/https links reach the user — hallucinated URLs are silently removed.
L4
Language Guardrails — Prompt + Structured Output
The system prompt forbids certainty language. The frontend renders validated JSON fields — never raw LLM text. A rogue sentence cannot surface in the UI under any circumstances.
CivicEase AI05 — Responsible AI
Human-in-the-Loop

Preparation, Not Determination

The AI is the guide. The human caseworker is the gatekeeper. That boundary is not configurable.

🚦
Clarification Gate
If any essential profile field is missing, the pipeline halts entirely and asks a targeted follow-up. No policy matching occurs on incomplete data.
📊
Confidence Scores
Agent 2 outputs a 0.0–1.0 confidence score per benefit match — presented as guidance only. Language like "may qualify" is enforced at the prompt level.
🏛️
Caseworker Decides
CivicEase AI delivers a preparation checklist. The government caseworker performs the final verification, determination, and authorization of benefits.
System never says: "You are approved" | Always says: "You may qualify"
